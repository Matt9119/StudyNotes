Display Filters — applied after capture
🚨
Display filters ≠ Capture filters (BPF). Display filters narrow what you see in an existing pcap. They use a different syntax from BPF and cannot produce a new pcap file. Use File → Export Specified Packets to save a filtered subset.

IPv6Layer 3 — fixed size40 bytes
TCPLayer 4 — minimum, no options20 bytes
UDPLayer 4 — fixed size8 bytes
ICMPEcho request/reply header8 bytes
DNSQuery/response header12 bytes (+ variable records)
ARPRequest/reply for IPv428 bytes (+ optional 36 byte trailer)
802.1Q VLAN tagAppended to Ethernet header+4 bytes
🧮 IPv4 payload calculation
IPv4 Payload = Total Length − (IHL × 4)
IHL (Internet Header Length) is a 4-bit field showing the
header size in 32-bit words — multiply by 4 to get bytes.
Standard IHL value = 5 → 5 × 4 = 20 bytes
Example: Total Length = 1500, IHL = 5
1500 − (5 × 4) = 1500 − 20 = 1480 bytes of payload
Max payload at standard MTU:
MTU 1500 − 20 (IP) = 1480 bytes
With TCP: 1500 − 20 (IP) − 20 (TCP) = 1460 bytes (MSS)
✂ Fragment offset calculation
Byte Position = Fragment Offset × 8
Fragment Offset is stored in 8-byte blocks.
Multiply the field value by 8 to get the byte
position in the original datagram.
Example: Offset field = 185
185 × 8 = 1480 byte offset
Max Fragment Offset at 1500 MTU:
(1500 − 20) / 8 = 185 blocks
IPv4 Flags field (3 bits):
Bit 0 — Reserved (always 0)
Bit 1 — DF (Don't Fragment)
Bit 2 — MF (More Fragments — 1 = more coming)
🧮 TCP payload calculation
TCP Payload = IP Total Length − IP Header − TCP Header
TCP Data Offset field (4 bits) = TCP header size in 32-bit words.
Multiply by 4 to get bytes (same logic as IPv4 IHL).
Standard Data Offset = 5 → 5 × 4 = 20 bytes
Example: IP Total Length = 1500, IHL = 5, TCP Data Offset = 5
1500 − 20 − 20 = 1460 bytes TCP payload
⚠ IPv4 vs IPv6 length field difference
IPv4: Total Length = Header + Payload
Payload = Total Length − (IHL × 4)
IPv6: Payload Length = Payload ONLY
IPv6 header is NOT included in the Payload Length field
Total size = Payload Length + 40 (fixed header)
⚠ Don't mix these up — the field names are
different and mean different things.

🤝 3-Way Handshake
ClientServer
── SYN ──────────────────────────→
←──────────────────────── SYN-ACK ──
── ACK ──────────────────────────→
Step 1 — Client sends SYN (Seq=x)
Step 2 — Server replies SYN-ACK (Seq=y, Ack=x+1)
Step 3 — Client sends ACK (Ack=y+1) → connection open
4-Way Teardown:
FIN → ACK → FIN → ACK
Either side can initiate with FIN
IPv4 Flags & Fragmentation Offset — combined interpretation
🧮
The IPv4 Flags field (3 bits) and Fragment Offset field (13 bits) are read together to determine a packet's fragmentation state. The two flag bits that matter are DF (Don't Fragment, bit 1) and MF (More Fragments, bit 2).
✂ Flags + Fragment Offset — what each combination means
Flags DF MF Frag Offset Meaning
0x00 0 0 0 Unfragmented packet — no fragmentation at all
0x02 1 0 0 DF set — device says "do not fragment this packet"
0x01 0 1 0 First fragment — more fragments follow, this is the start
0x01 0 1 > 0 Middle fragment — more coming, not the first or last
0x00 0 0 > 0 Last fragment — MF cleared, offset shows position
📌 Wireshark filter reference — flags
ip.flags == 0x00No flags setNormal unfragmented packet
ip.flags.df == 1Don't Fragment bit setPacket cannot be split
ip.flags.mf == 1More Fragments bit setFirst or middle fragment
ip.frag_offset == 0Offset is zeroFirst or only fragment
ip.frag_offset > 0Non-zero offsetMiddle or last fragment
ip.flags.mf == 1 && ip.frag_offset == 0MF=1, Offset=0First fragment only
ip.flags.mf == 0 && ip.frag_offset > 0MF=0, Offset>0Last fragment only
How to identify all fragments of one datagram
All fragments of the same original datagram share the same Identification field value — filter with ip.id == 0x1234 to group them.
🚨
DF + ICMP "Fragmentation Needed" = Path MTU Discovery in action. If a router can't forward a packet without fragmenting it but DF is set, it drops the packet and sends ICMP Type 3 Code 4 back to the sender. This is normal behavior — not an attack.

🔄 Conversion methods
Decimal → Hexadecimal
Repeatedly divide by 16. Remainders (bottom-up) = hex digits.
Example: 425 → hex
425 ÷ 16 = 26 remainder 9
26 ÷ 16 = 1 remainder A
1 ÷ 16 = 0 remainder 1
Read remainders bottom-up → 0x1A9
Hexadecimal → Decimal
Multiply each digit by 16position (right = position 0).
Example: 0x1A9 → decimal
9 × 16⁰ = 9 × 1 = 9
A × 16¹ = 10 × 16 = 160
1 × 16² = 1 × 256 = 256
Sum: 9 + 160 + 256 = 425
Hex → Binary
Each hex digit = exactly 4 binary bits. Look up each digit in the table.
0x1A9 → 0001 1010 1001
1=0001 · A=1010 · 9=1001

💡
Quick triage steps for an unknown pcap:
1. Statistics → Protocol Hierarchy — what protocols are present?
2. Statistics → Conversations — who is talking to whom, how much data?
3. Filter to the interesting IP/port with display filters
4. Right-click → Follow Stream to read the full exchange
5. Ctrl+F to search for keywords (passwords, credentials, commands) in packet data
⚠
Key things to check in the details pane:
• TTL — low TTL may indicate a spoofed source or traceroute scan
• DF flag set + ICMP "fragmentation needed" — Path MTU discovery in action
• Repeated RST packets — port scan or connection rejection
• Sequence number anomalies — possible injection or hijacking attempt
• Large DNS TXT records — possible DNS tunneling or exfiltration
🧮
Quick math recap:   IPv4 payload = ip.len − (ip.hdr_len × 4)  ·  TCP payload = ip.len − ip.hdr_len − tcp.hdr_len  ·  IPv6 payload = ipv6.plen (already excludes 40-byte header)  ·  Fragment byte position = ip.frag_offset × 8
