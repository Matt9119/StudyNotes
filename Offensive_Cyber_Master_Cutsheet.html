Initial Setup — Before You Start
📋
Initial info you'll be given: Attack Platform IP, Redirector IP, Initial Target IP. Configure everything below before beginning the engagement.
🐉 Kali
1
Open Kali and create tabs for each role (rename + color-code for readability): Kali, MSF (msfconsole), RDR (redirector), etc.
2
Configure the Kali proxy to use socks5 — edit /etc/proxychains4.conf. In vim: Shift+G to jump to bottom, insert mode, change socks4 → socks5.
vim /etc/proxychains4.conf
🔀 Redirector
1
Establish a combined SSH tunnel (dynamic + remote port forwarding simultaneously) from the RDR tab:
ssh -i <identity_key> -D <attack_platform_proxy_port> -R <ssh_server_IP>:<ssh_server_bind_port>:<attack_platform_IP>:<listening_port> <redirector_username>@<redirector_server_IP>
💣 Metasploit (msfconsole globals)
1
Set global variables so every module inherits them without re-typing:
setg lhost <redirector_ip> setg lport <redirector_port> setg ReverseListenerBindAddress <attack_platform_ip> setg ReverseListenerBindPort <attack_platform_port> setg ReverseAllowProxy true setg Proxies socks5:127.0.0.1:9050
2
View all currently-set global variables:
setg
🦊 Firefox Browser Proxy
1
Settings → search "proxy" → Manual Proxy Configuration.
2
SOCKS Host = 127.0.0.1, Port = 9050. No Proxy for 1.1.1.1.
3
Check "Proxy DNS when using SOCKS v5", then OK.
⛓️ Proxychains
1
Prefix any command run from your attack platform to mask your true IP and potentially bypass restrictive firewalls:
proxychains <command> <target>
🔓 John the Ripper
1
Crack hashed credentials:
john <hash_file>
2
You can explicitly force John to target NTLM/NT hashes as well as use previously cracked passwords stored in your john.pot file::
john --format=nt <hash_file> --loopback
3
Delete state files before cracking a new hash file, or John will resume the old session instead of starting fresh:
rm ~/.john/john.pot ~/.john/john.rec
📡 Scanline
1
Find sl.exe on your attack platform
locate sl.exe
2
Move sl.exe to your current directory to simplify the meterpreter upload command
mv <filepath to sl.exe> .

2Enumeration / Scanning
nmap — key flags
FlagPurpose
-sSSYN ("half-open") scan — fast, stealthier, needs root
-sTFull TCP connect scan — no root required, noisier
-sUUDP scan
-sVService/version detection
-sCRun default NSE scripts
-AAggressive: OS detect + version + scripts + traceroute
-OOS detection
-p-Scan all 65535 ports (or -p 1-1000 for a range)
-PnSkip host discovery — treat host as up (needed if ICMP is blocked)
-T0 – T5Timing template, 0=paranoid/slow, 5=insane/fast
-oA <name>Save output in all formats (.nmap/.xml/.gnmap)
--script=vulnRun NSE vulnerability-detection scripts
-v / -vvIncrease verbosity
Scanline — key flags verify against your lab build
FlagPurpose
-t <ports>TCP port(s) to scan, comma/range list
-u <ports>UDP port(s) to scan
-pDo not ping hosts before scanning (scan even if ping fails)
-bGrab service banners on open ports
-hHide results for hosts with no open ports
-c <n>Number of scanning threads
-o <file>Write output to a file

🛰️route add — pivoting through a session

ℹ
Once a route is added, other modules (scanners, exploits, proxies) can target hosts on that subnet as if directly reachable — this is what lets you pivot deeper into a network from one compromised box. Pair with set Proxies / auxiliary/server/socks_proxy if you want to route external tools (nmap, browser, etc.) through the same pivot.

Troubleshooting & General Tips
🩹First things to check
--help / -hCheck exact syntax/flags for the installed tool versionnmap -h  |  sl -?
man <cmd>Full manual page (Linux)man proxychains4
which / typeConfirm tool is installed & on PATHwhich nmap
Case sensitivityLinux paths/commands are case-sensitive; Windows generally isn't/Etc ≠ /etc
🌐Connectivity & pivoting
ping / Test-NetConnectionConfirm basic reachability firsttnc <host> -Port 445
Verify tunnel is upCheck the RDR tab's SSH session hasn't droppedre-run the ssh -D -R command
proxychains configConfirm socks5 + correct port match msf Proxies settingcat /etc/proxychains4.conf
LHOST/LPORT mismatchMost common reason a reverse shell never calls backre-check setg values
💡
General rule: when a command "does nothing," it's almost always syntax, PATH, a missing listener, or a proxy/tunnel that silently dropped — check those four before assuming the exploit itself is broken.
