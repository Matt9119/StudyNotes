Data Encapsulation & De-encapsulation

Encapsulation
Sender adds headers (and a trailer at L2) as data travels down the OSI stack.
De-encapsulation
Receiver strips headers and trailer as data travels up the OSI stack.

Data Link Layer — Layer 2
Frames, MACs & Protocols

PDU & key facts

PDU
Frames — adds both a header and a trailer (FCS) to data.
MAC address modification
Routers update the source and destination MAC addresses when forwarding packets between LAN interfaces — IP addresses stay the same.

Layer 2 devices

Switch (L2 only)
Connects multiple devices on a LAN. Reduces collision domains using packet switching. Routes by MAC address to physical port. Supports VLANs and PNAC (802.1X).
Multi-Layer Switch (L2 + L3)
All switch capabilities plus Layer 3 routing decisions.
Bridges & WAPs
Also operate at Layer 2. WAPs connect wireless clients to a wired network.

MAC address structure

Format
6 bytes written as 12 hex characters — e.g. AA:BB:CC:11:22:33
OUI (first 3 bytes)
Organizational Unique Identifier — identifies the manufacturer or vendor.
NIC/Serial (last 3 bytes)
Identifies the specific device.

Layer 2 protocols

ProtocolSizePurpose
802.3 (Ethernet) 14 bytes Most basic L2 header. Contains source MAC, destination MAC, and 2-byte Ethertype (identifies next layer protocol, e.g. IPv4).
802.1Q (VLAN) +4 bytes Appends a 4-byte VLAN tag to the Ethernet header for logical traffic separation over L2 devices.
Exam trap

Switches operate at Layer 2 only. Multi-Layer Switches operate at Layer 2 AND Layer 3. Routers operate at Layer 3. Know which device lives at which layer.

Network Layer — Layer 3
Packets, Routing & IP Addressing

Layer 3 key facts

PDU
Packets — integrity check (checksum) is built into the L3 header (unlike Frames which use a trailer).
Standards body
IETF defines Layer 3 standards using the RFC system. RFC 791 = IPv4. RFC 8200 = IPv6.
Frame stripping
When traffic leaves a LAN, L3 devices strip the frame to read the packet header and make routing decisions.

Layer 3 devices

Router
Connects multiple networks and Autonomous Systems (the internet). Builds routing tables dynamically using RIP or OSPF. Routes by IP address.
Multi-Layer Switch
Can make both L2 (MAC) and L3 (IP) routing decisions.
IPv4

IPv4 overview

Address size
32-bit / 4 bytes — written as 4 decimal octets 0–255 separated by periods. Ex: 192.168.0.1
Address space
232 unique addresses (~4.3 billion).
Header size
20 bytes (standard, without options).
Delivery model
Connectionless, best-effort — no guaranteed delivery, ordering, or duplicate prevention without TCP.
Private addressing
Supports internal routing via NAT and private IP ranges.
MAC mapping
Uses ARP (Address Resolution Protocol) to map IP → MAC.

IPv4 header structure

IPv4 vulnerabilities

Spoofing (Confidentiality / Availability)
Attacker forges packets from a trusted source address to bypass auth, launch DoS attacks, or redirect traffic.
Sniffing (Confidentiality)
Attacker intercepts and reads unencrypted packets to steal sensitive data.
IPv6

IPv6 overview

Address size
128-bit / 16 bytes — written as 8 groups of 4 hex characters separated by colons. Ex: 1111:2202:0333:0004:ffff:0000:0000:bbbb
Address space
2128 unique addresses — effectively unlimited.
Header size
40 bytes (fixed, simpler than IPv4).
MAC mapping
Uses NDP (Neighbor Discovery Protocol) instead of ARP.
Security
IPSec is mandatory when requested (optional in IPv4).
No NAT
True end-to-end connectivity — every device gets a globally unique address.

IPv6 address abbreviation rules

Rule 1
Drop leading zeros within each group. 0333 → 333
Rule 2
Replace the largest consecutive group of all-zero blocks with :: — only once.
Rule 3
Replace any other single all-zero group with 0.
Example
1111:2202:0333:0004:ffff:0000:0000:bbbb → 1111:2202:333:4:ffff::bbbb

IPv6 vulnerabilities

NDP misconfig (Confidentiality / Integrity)
Rogue DHCPv6 servers and misconfigured NDP can expose MAC addresses and reassign IPv6 addresses on a private network.
Higher misconfig risk (Integrity / Availability)
IPv6 has greater complexity, increasing the chance of device and security misconfiguration.

Exam trap

IPv6 headers do not contain a checksum — that was intentionally removed to reduce processing overhead. Error checking is handled by upper-layer protocols.

Fragmentation & Vulnerabilities

Fragmentation

What it is
Breaking data into chunks when payload exceeds the MTU (Maximum Transmission Unit). Receiver reassembles the chunks.
IPv4
Handled within the standard header using the Flags and Fragment Offset fields. Standard MTU = 1500 bytes → Fragment Offset in multiples of 8-byte blocks.
IPv6
Handled via an additional 8-byte fragmentation extension header — not in the base header.

Fragmentation vulnerabilities

Firewall evasion (Confidentiality / Integrity)
Firewalls may miss fragmented malware — it passes inspection in pieces and reassembles as malicious payload at the endpoint.
MITM fragment tampering (Integrity)
A man-in-the-middle attacker can alter individual fragments, corrupting a data transfer or injecting malicious content.
Fragment flood DoS (Availability)
Attacker spams fake fragments that can never be reassembled, exhausting memory buffers and causing a Denial of Service.

Transport Layer — Layer 4
Segments, Datagrams & Ports
PDU
Segments (TCP) or Datagrams (UDP) depending on the protocol used.
Checksum
Layer 4 contains its own checksum for error checking — like Layer 3.
Devices
Servers host services on fixed, well-known ports. Client systems (computers, mobile devices) connect to servers or make peer-to-peer connections.
Address type
Port numbers — identify specific services on a host.

Port ranges

RangeNamePurpose
0 – 1024Known / Well-Known PortsReserved for well-known services (HTTP, DNS, SSH, etc.)
1025 – 65535Ephemeral PortsDynamically assigned for short-lived client connections
Key takeaway

Servers listen on known ports. When a client connects, the OS assigns it an ephemeral port for that session. The pair (src IP:src port ↔ dst IP:dst port) uniquely identifies the connection.

TCP — Transmission Control Protocol

TCP overview

Connection type
Connection-oriented — establishes a session before sending data.
3-Way Handshake
SYN → SYN-ACK → ACK. Must complete before data transfer begins.
Reliability
Guarantees delivery, correct ordering, and protection from duplicates.
Data handling
Streams are broken into segments and reassembled by the receiver.
Header size
20 bytes.
Security note
A stream can be manipulated to evade detection by some firewalls and IDS.

TCP header fields

FieldPurpose
Source & Destination PortIdentifies sending and receiving services
Sequence NumberTracks order of segments for reassembly
Data OffsetIndicates where the data begins in the segment
ChecksumError detection for the header and data
DataVariable-length payload

TCP vulnerabilities

SYN-Flood (Availability)
Attacker spoofs legitimate IP addresses to spam SYN requests, overwhelming a server and denying service to legitimate users.
Connection Hijacking (Confidentiality)
Attacker eavesdrops on a session, sends TCP-RST to both client and server, then reconstructs the connections — inserting themselves into the stream.
TCP Veto (Confidentiality)
While eavesdropping, attacker spoofs the expected packet size and replaces the packet with a malicious payload, which the receiver accepts.
UDP — User Datagram Protocol

UDP overview

Connection type
Connectionless — no handshake, no session establishment.
Reliability
Unreliable — does not guarantee delivery, ordering, or duplicate protection.
Best used for
Real-time applications — gaming, streaming, VoIP — where speed matters more than accuracy.
Security
Like TCP, relies on upper-layer protocols (e.g. HTTPS) for security and error checking.
Header size
8 bytes — much smaller than TCP.

UDP header fields

FieldPurpose
Source & Destination PortIdentifies sending and receiving services
Message LengthTotal length of the UDP header + data
ChecksumOptional — basic error detection only
DataVariable-length payload

UDP vulnerabilities

Forged datagram DoS (Availability)
Attackers can forge UDP datagrams with spoofed source addresses to flood a target and cause a Denial of Service.

Exam trap

UDP is not inherently less secure than TCP — it's just simpler. Both rely on upper-layer protocols like HTTPS for encryption and security. The key difference is reliability vs speed, not security.

DNS — Domain Name System

DNS overview

Purpose
Resolves a Fully Qualified Domain Name (FQDN) to an IP address — or an IP back to a domain name.
Header size
12 bytes.

DNS message types

DNS ResponseServer replies with the resolved address

DNS vulnerabilities

DNS Exfiltration (Confidentiality)
Attacker encodes stolen data inside DNS queries to an external server, bypassing firewalls that allow DNS traffic.
DNS Tunneling (Confidentiality)
Attacker inserts a string within normal DNS communications to establish a covert remote connection or C2 channel.
DNS Flood (Availability)
Attacker spams a client with valid-looking DNS responses to cause a Denial of Service.
DNS Hijacking (Integrity)
Attacker modifies DNS server records to redirect legitimate domain lookups to a malicious IP address.
DHCP — Dynamic Host Configuration Protocol

DHCP overview

Purpose
Dynamically assigns (leases) temporary IP addresses to devices from a pool of valid IPs for the network.
Authentication
Lacks authentication — cannot verify the identity of the DHCP server or client.

DHCP vulnerabilities

Rogue DHCP Server (Confidentiality / Integrity)
An attacker sets up an unauthorized DHCP server that assigns clients a malicious gateway or DNS server, redirecting all traffic through the attacker.
Starvation Attack (Availability)
Attacker exhausts the DHCP address pool by requesting all available IPs, preventing legitimate devices from obtaining an address.
ARP — Address Resolution Protocol

ARP overview

Purpose
Dynamically maps a logical IPv4 address to a physical MAC address within a LAN using ARP requests and replies.
Header size
28 bytes. Optional trailer of 36 bytes.
Security weakness
No way to verify a MAC address is truly associated with an IP. Systems always update their ARP table when they receive an ARP reply — even unsolicited ones.

ARP header fields

FieldPurpose
Hardware TypeType of network (e.g. Ethernet)
Protocol TypeLayer 3 protocol (e.g. IPv4)
Source MACMAC address of the requesting device
Source IPIP address of the requesting device
Target MACMAC address being sought (blank in requests)
Target IPIP address being resolved

ARP vulnerabilities

ARP Cache Poisoning (Integrity / Availability)
Flooding a device with excessive unsolicited ARP replies corrupts its ARP cache and can cause a Denial of Service.
ARP Spoofing (Confidentiality / Integrity)
Attacker associates their MAC address with a legitimate IP address. Traffic intended for the legitimate host is instead sent to the attacker, enabling interception or modification (MITM).
Exam trap

ARP has no authentication. A device will accept and cache any ARP reply it receives — even if it never sent a request. This is the root cause of both ARP Cache Poisoning and ARP Spoofing attacks.

Operational Technology (OT)
OT & Critical Infrastructure
What is OT?
A broad range of programmable systems and devices that interact with or manage the physical environment — manufacturing, mechanical, hydraulic, aeronautic systems.
PPD-21
Presidential Policy Directive 21 defined 16 critical infrastructure (CI) sectors and established national policy to secure and maintain resilient CI.
OT-reliant sectors (13)
Chemical, Commercial Facilities, Critical Manufacturing, Dams, Defense Industrial Base, Emergency Services, Energy, Food & Agriculture, Government Facilities, Healthcare, Nuclear, Transportation, Water.
Non-OT sectors (3)
Communications, Financial, Information Technology.
Key takeaway

OT prioritizes availability and safety over confidentiality — the opposite priority order from traditional IT security.

SCADA
Supervisory Control & Data Acquisition
What it is
An ICS architecture providing centralized monitoring and control of geographically dispersed assets. Primary goal: centralized data acquisition.
Heartbeat networks
SCADA networks (L1/L2) send periodic pulses from the command center (SCADA server) to field controllers to verify operational status.

SCADA — 3 main components

Central Command Center
All servers running SCADA software.
Local Control Systems
Remotely located — directly control and automate field/remote-site equipment.
Communication Systems
Connect the central command center servers to all remote locations.

Specialized SCADA field hardware

DeviceFull nameHow it acts
PLCProgrammable Logic ControllerProgrammed — may act with or without instructions
RTURemote Terminal UnitActs primarily with instructions from control server
IEDIntelligent Electronic DeviceActs primarily without instructions (autonomous)
Distributed Control System (DCS)
Purpose
Controls production systems within the same geographical location (vs SCADA which spans geographic distances).
How it works
Uses a centralized supervisory control loop to mediate a group of localized controllers, each handling part of an entire production process.
Exam trap

SCADA = geographically dispersed assets (remote sites, pipelines, power grids). DCS = assets within the same location (a single factory floor).

OT Security

Threat landscape

Primary threat
Nation-state APTs — disrupting a major utility can severely impact a nation's population or economy.
Secondary threat
Cybercriminal organizations targeting CI for financial gain or disruption.
Attack impacts
Financial losses, stolen data, disruption to civilizational functions, and direct threats to human safety.
Expanding surface
The Industrial Internet of Things (IIoT) is continuously expanding OT attack surfaces.

Security challenges

ChallengeWhy it's hard
Availability RequirementsOutages must be planned weeks in advance — patching windows are rare
Change ManagementVulnerabilities are often left unpatched for extended periods
Resource ConstraintsMany legacy systems cannot perform encryption/decryption
Component LifecycleLong lifecycles mean insecure legacy systems remain in production
Managed SupportOT support typically comes from a single vendor
Component LocationRemote, isolated sites require physical and environmental security
Protocol SecurityMany OT protocols include no security features whatsoever

Security measures

  • Asset Inventory — identify and mitigate vulnerable devices
  • Vulnerability/Patch Management — update vulnerable software on schedule
  • IDS/IPS — monitor and control data transfer between IT and OT networks
  • Endpoint Detection & Response — invest in OT systems with security processing capability
  • Physical Access Control — restrict OT system access to privileged users only
  • Physical Security — secure and protect the physical systems and their environments
Berkeley Packet Filters (BPF)
Filtering Network Traffic
What is BPF?
Designed for the Linux kernel — enables granular packet processing using libpcap (Linux) and WinPcap (Windows) libraries. Can filter both live and captured traffic.
eBPF
Extended BPF — used by Windows and included in some Linux kernels. Many security tools support both BPF and eBPF.
Why use BPFs?
Acts as a raw interface with the Data Link layer — filter by traffic type, IP, port, or individual flags (SYN, ACK, etc.). Combined with threat intelligence, dramatically reduces analysis time.
